# Scope Document – Vocabulary Quiz Generator (Python Console)

## 1. ภาพรวมโครงการ (Project Overview)

โครงการนี้เป็นการพัฒนาโปรแกรมแบบ Console สำหรับสร้างแบบทดสอบคำศัพท์อัตโนมัติ (Vocabulary Quiz Generator) โดยโปรแกรมจะอ่านข้อมูลคำศัพท์จากไฟล์ภายนอก (เช่น .txt หรือ .csv) จากนั้นสุ่มคำถามแบบปรนัย 4 ตัวเลือก ให้ผู้ใช้ตอบและแสดงคะแนนเมื่อจบแบบทดสอบ

ระบบถูกออกแบบให้มีโครงสร้างเรียบง่าย เน้นความถูกต้องของตรรกะ (Logic), การจัดการข้อมูลจากไฟล์, การสุ่มคำถาม และการรายงานผลคะแนนรวมของผู้ใช้

---

## 2. วัตถุประสงค์ของระบบ (Objectives)

1. เพื่อให้ผู้ใช้สามารถทำแบบทดสอบคำศัพท์จากไฟล์คำศัพท์ที่เตรียมไว้ล่วงหน้าได้อย่างสะดวก
2. เพื่อฝึกการอ่าน/ประมวลผลข้อมูลจากไฟล์ภายนอก และการสุ่มคำถามแบบมีตัวเลือก
3. เพื่อแสดงคะแนนรวมและสรุปคำตอบหลังจบแบบทดสอบ เพื่อช่วยให้ผู้ใช้เห็นจุดที่ควรทบทวน
4. เพื่อใช้เป็นกรณีศึกษาแสดงกระบวนการพัฒนา Software ตาม SDLC ตั้งแต่ Requirement → Design → Implementation → Testing → Deployment

---

## 3. ผู้มีส่วนเกี่ยวข้องและผู้ใช้งาน (Stakeholders & Users)

- **นักศึกษา (Developer)**  
  - ผู้พัฒนาระบบ, ออกแบบ, เขียนโค้ด และจัดทำเอกสาร

- **ผู้ใช้ปลายทาง (End User)**  
  - ผู้ใช้ทั่วไปที่ต้องการทำแบบทดสอบคำศัพท์ผ่านหน้าจอ Console

- **อาจารย์ผู้สอน (Instructor)**  
  - ผู้ตรวจประเมินคุณภาพโค้ด เอกสาร และการออกแบบตามเกณฑ์ที่กำหนด

---

## 4. ขอบเขตของระบบ (System Scope)

### 4.1 ฟังก์ชันที่อยู่ในขอบเขต (IN SCOPE)

1. อ่านข้อมูลคำศัพท์จากไฟล์ภายนอก (เช่น .txt หรือ .csv)  
   - โครงสร้างไฟล์จะถูกกำหนดรูปแบบที่ชัดเจน เช่น  
     `word,correct_answer,choice1,choice2,choice3,choice4`

2. แสดงเมนูหลักบน Console  
   - เช่น เริ่มทำแบบทดสอบ / เปลี่ยนไฟล์คำศัพท์ / ออกจากโปรแกรม

3. สุ่มคำถามจากรายการคำศัพท์  
   - สุ่มลำดับคำถาม  
   - สุ่มลำดับตัวเลือกคำตอบ (4 ตัวเลือก)

4. รับคำตอบจากผู้ใช้ผ่านคีย์บอร์ด (เช่น กด 1–4 หรือ A–D)

5. ตรวจสอบคำตอบว่าถูก/ผิด  
   - เพิ่มคะแนนเมื่อผู้ใช้ตอบถูก  
   - เก็บข้อมูลคำถามที่ตอบผิด

6. แสดงผลคะแนนรวมเมื่อจบแบบทดสอบ  
   - จำนวนข้อที่ทำ  
   - จำนวนข้อที่ตอบถูก  
   - จำนวนข้อที่ตอบผิด  
   - คิดเป็นเปอร์เซ็นต์คะแนน

7. แสดงสรุปรายการคำถามที่ตอบผิด (อย่างน้อย: คำศัพท์ + คำตอบที่ถูกต้อง)

8. จัดการข้อผิดพลาดพื้นฐาน  
   - ไฟล์ไม่พบ / เปิดไม่ได้  
   - รูปแบบข้อมูลในไฟล์ไม่ถูกต้องบางส่วน (เช่น ข้ามบรรทัดที่ไม่สมบูรณ์)  
   - ผู้ใช้กรอกตัวเลือกผิดรูปแบบ (เช่น กดเลขนอกช่วง 1–4)

---

### 4.2 ฟังก์ชันที่อยู่นอกขอบเขต (OUT OF SCOPE)

1. ไม่มีระบบจัดการผู้ใช้ (User Login / Register)
2. ไม่มีการบันทึกประวัติการทำแบบทดสอบของผู้ใช้ลงฐานข้อมูล
3. ไม่มีการแสดงกราฟหรือรายงานเชิงสถิติขั้นสูง (เช่น กราฟเส้น/แท่ง)
4. ไม่มี UI แบบกราฟิก (GUI) หรือ Web Interface (ระบบนี้เป็น Console Application เท่านั้น)
5. ไม่มีการดึงคำศัพท์จากแหล่งข้อมูลออนไลน์ หรือเชื่อมต่อ API ภายนอก
6. ไม่มีการสร้างหรือแก้ไขไฟล์คำศัพท์ผ่านโปรแกรม (ไฟล์คำศัพท์เตรียมไว้ล่วงหน้าด้วยมือ)

---

## 5. ฟังก์ชันหลักของระบบ (High-Level Features)

1. **Import Vocabulary File**  
   - ผู้ใช้ระบุชื่อไฟล์คำศัพท์  
   - ระบบอ่านไฟล์และโหลดคำศัพท์เข้าสู่หน่วยความจำ

2. **Start Quiz**  
   - ระบบสุ่มคำถามตามจำนวนที่กำหนด (เช่น ทำทุกคำ หรือจำกัดจำนวนข้อ)  
   - แสดงคำถามทีละข้อและตัวเลือก 4 ตัวเลือก  
   - รับคำตอบจากผู้ใช้

3. **Answer Evaluation & Scoring**  
   - ตรวจสอบคำตอบทันทีต่อข้อ หรือจะสรุปตอนท้าย (ตามแบบที่ออกแบบไว้)  
   - เก็บคะแนนรวม และรายการข้อที่ตอบผิด

4. **Result Summary**  
   - แสดงคะแนนรวม  
   - แสดงจำนวนข้อถูก/ผิด  
   - แสดงรายการคำศัพท์ที่ตอบผิด พร้อมคำตอบที่ถูกต้อง

5. **Basic Error Handling**  
   - แจ้งเตือนผู้ใช้เมื่อไฟล์ไม่พบ หรือรูปแบบไฟล์ไม่ถูกต้อง  
   - แจ้งเตือนเมื่อกรอกตัวเลือกผิด และให้กรอกใหม่

---

## 6. ข้อจำกัดและสมมติฐาน (Constraints & Assumptions)

### ข้อจำกัด (Constraints)

1. ระบบเป็น **Console Application** พัฒนาโดยใช้ภาษา **Python** เท่านั้น
2. ข้อมูลคำศัพท์ต้องอยู่ในรูปแบบไฟล์ข้อความ (Text/CSV) ที่ตรงตามรูปแบบที่กำหนด
3. ระบบจะรันบนเครื่องผู้ใช้ (Local Machine) ไม่รองรับการใช้งานผ่าน Web หรือ Mobile

### สมมติฐาน (Assumptions)

1. ผู้ใช้มีไฟล์คำศัพท์ที่ถูกต้องตามรูปแบบ (เช่น word + คำแปล + ตัวเลือก) เตรียมไว้แล้ว
2. ผู้ใช้สามารถใช้งาน Command Line / Terminal เบื้องต้นได้
3. จำนวนคำศัพท์ในไฟล์เพียงพอสำหรับสร้างแบบทดสอบหลายข้อ

---

## 7. เกณฑ์ความสำเร็จของระบบ (Success Criteria)

1. ระบบสามารถอ่านไฟล์คำศัพท์และสร้างคำถามแบบ 4 ตัวเลือกได้ถูกต้อง
2. ผู้ใช้สามารถทำแบบทดสอบจนจบและได้รับคะแนนสรุป
3. ระบบสามารถจัดการข้อผิดพลาดพื้นฐานได้ โดยไม่ทำให้โปรแกรมล้ม (ไม่ crash)
4. เอกสารทุกส่วน (Requirements, Design, Testing, User Manual) จัดทำครบตาม SDLC Checklist
5. Source Code อยู่ใน GitHub Public Repository พร้อม README.md และโครงสร้างโฟลเดอร์ตามที่กำหนด
